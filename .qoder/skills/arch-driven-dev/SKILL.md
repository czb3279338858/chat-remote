# 架构驱动开发技能

## 概述

本技能用于指导基于架构设计文档和需求文档进行系统开发，采用"计划先行、分步实施、用户确认"的开发模式。

核心原则：**编码任务需先说明思路，与用户讨论确认后再编写**。

## 工作流程

### 阶段1：文档分析

1. **读取需求文档**（`需求.md`）
   - 提取核心功能需求
   - 识别指令集和异常规则
   - 记录全局约束条件

2. **读取架构设计文档**（`docs/架构设计.md`）
   - 理解分层架构设计
   - 识别核心模块和接口
   - 明确目录结构设计
   - 了解设计模式应用

3. **分析输出**
   - 列出所有需要实现的功能点
   - 识别依赖关系和实现顺序
   - 标注风险点和注意事项

### 阶段2：制定开发计划

1. **任务分解**
   - 按模块分解任务
   - 确定模块间依赖顺序
   - 识别可并行开发的部分

2. **生成开发计划文档**

开发计划必须包含以下内容：

```markdown
# 开发计划

## 项目概述
- 项目名称：xxx
- 技术栈：xxx
- 架构模式：xxx

## 开发模式说明

### 核心原则：先澄清后实现

本计划严格执行**"编码任务需先说明思路，与用户讨论确认后再编写"**的开发模式：

```
编码任务执行流程：
1. 向用户说明实现思路
   - 文件职责
   - 关键接口/类型
   - 实现要点
   - 依赖关系

2. 与用户讨论确认
   - 讨论设计方案
   - 根据反馈调整
   - 获得明确确认

3. 用户确认后编写代码
```

### 非编码任务

以下任务可直接执行，无需说明思路：
- 项目初始化（package.json、tsconfig.json等）
- 依赖安装
- 配置文件创建
- 目录结构创建

## 任务清单

### 阶段1：xxx
- [ ] 任务1
- [ ] 任务2

### 阶段2：xxx
- [ ] 任务3
- [ ] 任务4

## 依赖关系图
[模块依赖关系说明]

## 当前进度
- 当前阶段：xxx
- 下一个任务：xxx
```

3. **计划确认**
   - 向用户展示完整计划
   - 根据用户反馈调整
   - 获得用户确认后开始执行

### 阶段3：逐步实施

**任务执行循环：**

```
while 还有未完成任务:
    1. 选择下一个任务
    
    2. 如果是编码任务:
       a. 向用户说明实现思路
          - 简要说明该文件的职责
          - 描述关键接口/类型定义
          - 说明实现要点和设计决策
          - 列出依赖关系
       
       b. 与用户讨论并确认方案
          - 讨论设计方案的合理性
          - 根据反馈调整方案
          - 获得用户明确确认
       
       c. 用户确认后编写代码
          - 根据确认的方案编写代码
          - 遵循架构文档中的规范
          - 添加必要的注释
    
    3. 如果是配置/初始化任务:
       - 直接执行
    
    4. 验证任务完成
    
    5. 更新进度
```

### 阶段4：验证与迭代

1. **任务完成验证**
   - 检查代码是否符合架构设计
   - 验证接口定义是否一致
   - 确认错误处理是否完善

2. **进度更新**
   - 更新任务清单状态
   - 记录已完成的功能
   - 标记阻塞或风险项

3. **用户同步**
   - 汇报当前进度
   - 说明下一步计划
   - 确认是否需要调整优先级

## 关键原则

### 1. 先澄清后实现原则
- 任何编码工作前先向用户说明实现思路
- 与用户讨论并确认方案
- 获得明确确认后再编写代码
- 方案变更需重新确认

### 2. 单文件聚焦原则
- 一次只处理一个文件的编写
- 完成当前文件后再进入下一个
- 避免多文件并行编写导致的混乱

### 3. 架构一致性原则
- 严格遵循架构设计文档的分层结构
- 保持接口定义与文档一致
- 使用文档中指定的设计模式

### 4. 用户确认原则
- 关键决策点必须获得用户确认
- 计划变更需重新确认
- 用户有权调整任务优先级

## 异常处理

### 文档缺失或不完整
1. 检查是否存在 `需求.md` 和 `docs/架构设计.md`
2. 如果缺失，询问用户文档位置
3. 如果不完整，基于已有信息制定初步计划，并标注不确定性

### 架构与需求冲突
1. 记录冲突点
2. 优先遵循需求文档的功能要求
3. 向用户说明冲突并寻求决策

### 技术栈不明确
1. 从架构文档推断技术栈
2. 询问用户确认
3. 基于确认的技术栈制定计划

## 输出规范

### 计划文档位置
- 开发计划：`docs/开发计划.md`（必须包含"先澄清后实现"的流程说明）

### 代码规范
- 遵循架构文档中的命名规范
- 使用架构文档中定义的接口
- 实现文档中指定的错误处理策略

## 使用示例

**用户输入**：
```
根据需求和架构文档开始开发
```

**执行流程**：
1. 读取 `需求.md` 和 `docs/架构设计.md`
2. 分析并生成开发计划（**必须包含"先澄清后实现"流程说明**）
3. 展示计划给用户确认
4. 按顺序执行每个任务
5. **编码任务先说明思路，与用户讨论确认后再编写**
6. 定期同步进度
