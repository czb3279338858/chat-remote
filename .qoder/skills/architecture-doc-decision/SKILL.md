# 架构文档决策与生成

## 触发条件

当以下情况出现时，使用此 skill：
- 用户要求创建架构设计文档
- 项目已有需求文档但不确定是否需要架构文档
- 已有架构文档但需求发生变化，需要评估是否重新生成
- 新项目启动需要架构设计

## 核心原则

> **架构设计是一个有机整体，各章节之间存在内在关联。当任何部分需要修改时，必须重新生成完整的架构文档。**

### 禁止行为
- 直接编辑现有架构文档的某个章节进行局部修改
- 在现有文档上打补丁式更新

### 强制要求
- 一旦确定需要修改架构设计，必须完整重新生成整个文档
- 确保各章节之间的一致性和协调性
- 确保架构设计的完整性和连贯性

## 决策流程

### 步骤1：检查现有文件

1. 检查是否存在 `docs/架构设计.md` 或类似架构文档
2. 检查是否存在 `docs/需求.md` 或类似需求文档
3. 记录文件路径和最后修改时间

### 步骤2：评估架构必要性

如果**不存在**架构文档，分析需求复杂度，满足以下任一条件则需要架构文档：

| 条件 | 说明 |
|------|------|
| 多模块耦合 | 系统包含3个及以上核心模块 |
| 多实例/类型 | 需要管理多种类型实体或实例 |
| 复杂指令集 | 命令/接口数量超过5个 |
| 多运行模式 | 存在多种运行状态或生命周期 |
| 复杂数据流 | 数据在多个组件间流转处理 |
| 扩展性要求 | 需要支持后续功能扩展 |

### 步骤3：评估现有架构是否过时（如已存在）

对比现有架构文档与需求文档，检查以下差异：

```
检查清单：
- [ ] 需求中的功能模块是否都在架构中有对应设计
- [ ] 架构中的模块职责是否与需求一致
- [ ] 需求变更是否导致架构设计需要调整
- [ ] 数据流描述是否与需求匹配
- [ ] 异常处理策略是否覆盖所有需求场景
- [ ] 目录结构设计是否完整且合理
```

**重新生成条件**（满足任一）：
- 需求新增核心功能模块
- 原有架构设计无法支持新需求
- 架构中30%以上内容与需求不匹配
- 用户明确要求重新设计架构
- 需要对现有架构设计进行修改

## 架构文档标准结构

如需生成，使用以下九章结构：

```markdown
# 架构设计文档

## 1. 系统概述
- 核心目标
- 关键术语定义
- 设计原则
- 测试驱动开发原则（如适用TDD）

## 2. 分层架构设计
- 分层架构图
- 各层职责定义
- 层间依赖规则

## 3. 核心模块设计
- 模块划分与职责
- 模块间交互关系
- 关键接口定义

## 4. 数据流分析
- 核心业务流程数据流
- 状态转换图
- 数据持久化策略

## 5. 设计模式应用
- 使用的模式及场景
- 模式实现说明

## 6. 扩展性设计
- 扩展点识别
- 新增模块/类型指南

## 7. 目录结构设计
- 项目根目录结构
- 源代码目录组织（按模块/分层）
- 测试目录结构（单元测试与实现同级，集成/E2E测试集中管理）
- 配置文件目录
- 文档目录结构
- 目录命名规范

## 8. 关键约束与约定
- 命名规范
- 错误处理策略
- 性能约束

## 9. 可测试性设计（如适用TDD）
- 依赖注入规范
- 接口隔离原则
- 测试替身策略
- 测试分层策略
```

## 第7章：目录结构设计要求

### 7.1 项目根目录结构

提供完整的目录树，示例格式：

```
project-root/
├── src/                    # 源代码目录
│   ├── domain/
│   │   └── instance/
│   │       ├── instance-manager.ts      # 实现
│   │       └── instance-manager.test.ts # 单元测试（与实现同级）
│   └── ...
├── tests/                  # 集成测试和端到端测试目录
│   ├── integration/        # 集成测试
│   └── e2e/                # 端到端测试
├── docs/                   # 文档目录
├── config/                 # 配置文件目录
├── scripts/                # 脚本工具目录
├── assets/                 # 静态资源目录
├── package.json            # 项目配置
└── README.md               # 项目说明
```

### 7.2 源代码目录组织

- 按模块划分：每个核心模块对应一个子目录
- 或按分层划分：controller/、service/、repository/ 等
- 避免所有文件放在同一层级

### 7.3 测试目录组织规范

#### 单元测试（与实现同级）

- **位置**：与对应的实现文件放在同一目录
- **命名**：`{filename}.test.ts` 或 `{filename}.spec.ts`
- **范围**：单个类/函数的独立测试
- **特点**：快速、隔离、不依赖外部服务

```
src/
├── domain/
│   └── instance/
│       ├── instance-manager.ts          # 实现
│       ├── instance-manager.test.ts     # 单元测试
│       ├── instance.ts                  # 实现
│       └── instance.test.ts             # 单元测试
```

#### 集成测试（集中管理）

- **位置**：`tests/integration/` 目录
- **命名**：`{feature}-integration.test.ts`
- **范围**：多个模块协作测试
- **特点**：验证模块间交互

```
tests/
└── integration/
    ├── instance-lifecycle.test.ts       # 实例生命周期集成测试
    └── command-flow.test.ts             # 命令流程集成测试
```

#### 端到端测试（集中管理）

- **位置**：`tests/e2e/` 目录
- **命名**：`{scenario}.e2e.test.ts`
- **范围**：完整业务流程测试
- **特点**：模拟真实用户场景

```
tests/
└── e2e/
    └── full-workflow.e2e.test.ts        # 完整工作流程测试
```

### 7.4 目录命名规范

- 使用小写字母
- 多单词使用短横线连接（kebab-case）
- 保持命名简洁且具有描述性

### 7.5 各目录职责说明

为每个目录提供清晰的职责描述，说明该目录存放什么类型的文件。

| 目录 | 存放内容 | 示例 |
|------|----------|------|
| `src/**/*.test.ts` | 单元测试（与实现同级） | `instance-manager.test.ts` |
| `tests/integration/` | 集成测试 | 模块间交互测试 |
| `tests/e2e/` | 端到端测试 | 完整业务流程测试 |

## TDD 设计原则

当项目采用测试驱动开发（TDD）时，架构设计需遵循以下原则：

### 1. 接口先行原则

- 先定义接口契约，再写测试，最后实现
- 接口设计即测试的契约，确保可测试性

### 2. 依赖注入原则

- 所有依赖通过构造函数注入
- 禁止在类内部直接实例化依赖
- 依赖必须是接口类型，而非具体类

### 3. 接口隔离原则

- 避免大接口，拆分为小接口
- 每个接口只定义一个职责领域
- 便于 Mock 和 Stub

### 4. 无全局状态原则

- 避免使用单例模式（除非必要）
- 避免静态方法和全局变量
- 确保测试之间相互隔离

### 5. 测试目录组织原则

- **单元测试**：与实现文件同级，便于查找和维护
- **集成测试**：放在 `tests/integration/`，验证模块协作
- **端到端测试**：放在 `tests/e2e/`，验证完整业务流程

## 执行步骤

### 场景A：首次生成架构文档

1. 读取需求文档
2. 创建 `docs/` 目录
3. 生成 `docs/架构设计.md`
4. 按九章结构填充内容
5. **目录结构设计要求**：
   - 提供完整的项目目录树（使用 tree 格式或层级列表）
   - 说明每个目录的职责和存放内容
   - 目录命名遵循统一规范（小写、短横线连接）
   - 源代码按模块或分层组织，避免扁平化
   - 区分源代码、测试、配置、文档等不同类型文件
   - **测试目录组织**：
     - 单元测试与实现文件放在同一目录（便于查找和维护）
     - 集成测试和端到端测试放在 `tests/` 目录下分类管理
6. **TDD 支持（如用户要求）**：
   - 在"设计原则"中增加测试驱动开发原则
   - 在"关键接口定义"中确保接口可测试（依赖注入、接口隔离）
   - 在"可测试性设计"章节中定义：
     - 依赖注入规范（构造函数注入、禁止内部实例化）
     - 接口隔离原则（小接口优于大接口）
     - 测试替身策略（Mock/Stub 使用场景）
     - 测试分层策略（单元/集成/E2E 的比例和范围）
7. 确保抽象设计与实现细节分离

### 场景B：评估并重新生成

1. 读取现有架构文档和需求文档
2. 逐条对比检查清单
3. 如需要重新生成：
   - 备份原有架构文档（可选）
   - **必须完整重新生成整个架构文档，禁止局部修改**
   - 按场景A流程重新生成
   - 在文档中标注版本变更说明

## 输出要求

1. **决策结果必须明确**：需要生成/不需要生成/需要重新生成
2. **理由充分**：说明判断依据和关键发现
3. **架构文档抽象性**：区分"抽象设计"与"具体实现"
4. **目录结构完整性**：必须包含完整的目录树和各目录职责说明
5. **测试目录规范**：明确单元测试与实现同级，集成/E2E测试集中管理
6. **可扩展性**：预留未来功能扩展的设计空间
7. **TDD 支持**（如适用）：包含可测试性设计章节，定义依赖注入、接口隔离等规范
8. **完整性保证**：一旦确定需要修改架构设计，必须完整重新生成整个文档，禁止局部修改
