# 架构设计文档

## 1. 系统概述

### 1.1 核心目标
通过手机聊天软件，统一管理 PC 上**多种交互式/非交互式实例**，实现指令简洁、异常提示明确、全程无鉴权、纯透传交互。

### 1.2 关键术语定义

| 术语 | 定义 |
|------|------|
| 实例（Instance） | 被管理的运行实体，如 qodercli、node、python、ssh 等进程 |
| 实例类型（Type） | 实例的分类标识，每种类型有特定的启动参数和运行模式 |
| 交互式实例 | 启动后持续运行，支持多次接收输入、返回输出的实例 |
| 非交互式实例 | 启动后自动运行、输出结果、自动结束的实例 |
| 运行状态 | 实例的当前状态：运行中（空闲/忙碌）、已退出 |
| 默认实例 | 通过 `/use ID` 设置的当前操作目标实例 |

### 1.3 设计原则

1. **单一职责原则（SRP）**：每个模块只负责一个明确的职责领域
2. **开闭原则（OCP）**：对扩展开放（新增实例类型），对修改关闭
3. **依赖倒置原则（DIP）**：高层模块依赖抽象接口，不依赖具体实现
4. **纯透传原则**：消息不做业务处理，直接转发给目标实例
5. **无状态设计**：系统不维护用户会话状态，仅管理实例生命周期

### 2.2 各层职责定义

| 层级 | 职责 | 核心组件 |
|------|------|----------|
| 接入层 | 负责与外部系统（飞书）通信，消息收发 | 飞书消息适配器 |
| 应用层 | 命令解析、路由、格式化响应 | 命令解析器、路由器、响应格式化器 |
| 领域层 | 核心业务逻辑，实例生命周期管理 | 实例管理器、实例抽象、状态管理 |
| 基础设施层 | 技术实现细节，外部依赖封装 | 终端适配器、类型注册表、日志存储 |

### 2.3 层间依赖规则

1. **单向依赖**：上层可以依赖下层，下层不能依赖上层
2. **跨层禁止**：不允许跨层调用（如接入层直接调用领域层）
3. **接口依赖**：通过接口定义层间契约，实现解耦
4. **依赖注入**：所有依赖通过构造函数注入，禁止内部实例化

### 3.2 模块间交互关系

| 调用方向 | 交互内容 | 触发场景 |
|----------|----------|----------|
| 飞书模块 → 命令模块 | 原始消息 | 收到用户消息 |
| 命令模块 → 实例模块 | 结构化命令 | 解析出有效指令 |

| 实例模块 → 命令模块 | 执行结果 | 操作完成 |
| 命令模块 → 飞书模块 | 响应消息 | 需要回复用户 |

## 4. 数据流分析

### 4.1 核心业务流程数据流

#### 4.1.1 创建实例流程（`/new`）

```
用户 ──► 飞书 ──► 飞书适配器 ──► 命令解析器 ──► 命令路由器
                                              │
                                              ▼
用户 ◄── 飞书 ◄── 飞书适配器 ◄── 响应格式化器 ◄── 实例管理器
                                              ▲
                                              │
                                         终端适配器
                                              │
                                              ▼
                                         创建进程
```

**数据转换**：
1. 飞书消息 → `IncomingMessage`（用户ID、内容）
2. 内容 → `Command`（命令名、参数数组）
3. 参数 → `InstanceConfig`（类型、备注、启动参数）
4. 进程句柄 → `IInstance` 对象
5. 结果 → `CommandResult` → 文本响应

#### 4.1.2 发送消息流程（`/ID 内容` / 纯文本）

```
用户 ──► 飞书 ──► 飞书适配器 ──► 命令解析器 ──► 实例管理器.sendTo()
                                                        │
                                                        ▼
用户 ◄── 飞书 ◄── 飞书适配器 ◄── 响应格式化器 ◄──── 实例.send()
                                                        ▲
                                                        │
                                                   终端适配器
                                                        │
                                                        ▼
                                                   进程.stdin
                                                        │
                                                        ▼
                                                   进程.stdout ──► 用户
```

**数据转换**：
1. 消息内容 → 写入进程 stdin
2. 进程 stdout → 字符串输出
3. 输出 → 飞书文本消息

#### 4.1.3 实例退出流程

```
进程退出 ──► 终端适配器 ──► 实例.onExit() ──► 实例管理器
                                                  │
                    ┌─────────────────────────────┘
                    ▼
              从运行列表移除
                    │
                    ▼
              写入日志存储
                    │
                    ▼
              清理资源
```

## 5. 设计模式应用

### 5.1 策略模式（Strategy Pattern）

**应用场景**：不同类型的实例有不同的创建和执行策略

**实现方式**：
```typescript
// 策略接口
interface IInstanceStrategy {
  readonly mode: RunMode;
  create(params: string[]): Promise<IInstance>;
  validateParams(params: string[]): boolean;
}

// 具体策略
class QoderCliStrategy implements IInstanceStrategy { ... }
class NodeStrategy implements IInstanceStrategy { ... }
class PythonStrategy implements IInstanceStrategy { ... }
class SshStrategy implements IInstanceStrategy { ... }

// 上下文
class InstanceFactory {
  private strategies = new Map<string, IInstanceStrategy>();
  
  register(type: string, strategy: IInstanceStrategy): void {
    this.strategies.set(type, strategy);
  }
  
  create(type: string, params: string[]): Promise<IInstance> {
    const strategy = this.strategies.get(type);
    if (!strategy) throw new Error(`不支持的类型: ${type}`);
    return strategy.create(params);
  }
}
```

### 5.2 工厂模式（Factory Pattern）

**应用场景**：统一创建不同类型的实例对象

**实现方式**：
```typescript
// 抽象工厂
interface IInstanceFactory {
  create(config: InstanceConfig): Promise<IInstance>;
}

// 具体工厂
class InteractiveInstanceFactory implements IInstanceFactory { ... }
class NonInteractiveInstanceFactory implements IInstanceFactory { ... }

// 使用
class InstanceManager {
  private interactiveFactory = new InteractiveInstanceFactory();
  private nonInteractiveFactory = new NonInteractiveInstanceFactory();
  
  async create(config: InstanceConfig): Promise<IInstance> {
    const factory = config.mode === RunMode.INTERACTIVE 
      ? this.interactiveFactory 
      : this.nonInteractiveFactory;
    return factory.create(config);
  }
}
```

### 5.3 观察者模式（Observer Pattern）

**应用场景**：监听实例进程的输出和退出事件

**实现方式**：
```typescript
interface IInstanceProcess {
  // 订阅输出
  onOutput(handler: (data: string) => void): void;
  // 订阅退出
  onExit(handler: (code: number, reason: string) => void): void;
}

class InstanceProcess implements IInstanceProcess {
  private outputHandlers: ((data: string) => void)[] = [];
  private exitHandlers: ((code: number, reason: string) => void)[] = [];
  
  onOutput(handler: (data: string) => void): void {
    this.outputHandlers.push(handler);
  }
  
  private emitOutput(data: string): void {
    this.outputHandlers.forEach(h => h(data));
  }
}
```

### 5.4 命令模式（Command Pattern）

**应用场景**：封装所有用户指令为对象，支持统一处理和扩展

**实现方式**：
```typescript
// 命令接口
interface ICommand {
  readonly name: string;
  readonly args: string[];
  execute(context: CommandContext): Promise<CommandResult>;
}

// 具体命令
class HelpCommand implements ICommand { ... }
class NewCommand implements ICommand { ... }
class ListCommand implements ICommand { ... }
class KillCommand implements ICommand { ... }

// 调用者
class CommandInvoker {
  private commands = new Map<string, new () => ICommand>();
  
  register(name: string, CommandClass: new () => ICommand): void {
    this.commands.set(name, CommandClass);
  }
  
  async invoke(name: string, args: string[], context: CommandContext): Promise<CommandResult> {
    const CommandClass = this.commands.get(name);
    if (!CommandClass) {
      return { success: false, message: '未知命令' };
    }
    const command = new CommandClass();
    command.args = args;
    return command.execute(context);
  }
}
```

## 6. 扩展性设计

### 6.1 扩展点识别

| 扩展点 | 扩展方式 | 示例 |
|--------|----------|------|
| 新增实例类型 | 实现 `IInstanceStrategy` 并注册 | 添加 `docker`、`powershell` 类型 |
| 新增命令 | 实现 `ICommand` 并注册 | 添加 `/status` 命令 |
| 更换消息平台 | 实现 `IFeishuAdapter` 接口 | 适配微信、钉钉等 |
| 更换存储 | 实现 `ILogStorage` 接口 | 使用数据库存储日志 |
| 自定义响应格式 | 实现 `IResponseFormatter` | 添加 Markdown 格式支持 |

### 6.2 新增实例类型指南

要添加一个新的实例类型（如 `docker`），需要：

1. **创建策略类**：
```typescript
class DockerStrategy implements IInstanceStrategy {
  readonly mode = RunMode.NON_INTERACTIVE;
  
  validateParams(params: string[]): boolean {
    return params.length >= 1; // 至少需要一个镜像名
  }
  
  async create(params: string[]): Promise<IInstance> {
    const process = await terminalAdapter.spawn('docker', 
      ['run', '--rm', ...params]);
    return new NonInteractiveInstance(process);
  }
}
```

2. **注册类型**：
```typescript
instanceFactory.register('docker', new DockerStrategy());
typeRegistry.register({
  type: 'docker',
  description: '运行 Docker 容器',
  mode: RunMode.NON_INTERACTIVE,
  paramsDescription: '镜像名 [参数...]',
  exampleParams: 'ubuntu:latest echo hello'
});
```

3. **无需修改现有代码**，系统自动支持新类型的 `/new`、`/kill` 等操作

### 7.2 源代码目录组织

源代码按**分层架构**组织：

- `adapter/` - 接入层：处理外部系统交互（飞书）
- `application/` - 应用层：命令解析、路由、格式化
- `domain/` - 领域层：核心业务逻辑、实例管理
- `infrastructure/` - 基础设施层：技术实现、外部依赖
- `shared/` - 共享工具：跨层使用的工具函数、常量

### 7.3 目录命名规范

- **小写字母**：所有目录名使用小写
- **短横线连接**：多单词使用 kebab-case（如 `feishu-adapter`）
- **单数优先**：目录名使用单数形式（如 `command/` 而非 `commands/`）
- **语义明确**：目录名清晰表达其职责

### 7.4 测试目录组织规范

#### 单元测试（与实现同级）

- **位置**：与对应的实现文件放在同一目录
- **命名**：`{filename}.test.ts`
- **范围**：单个类/函数的独立测试
- **特点**：快速、隔离、不依赖外部服务

示例：
```
src/domain/instance/
├── instance.ts              # 实现
├── instance.test.ts         # 单元测试
├── instance-manager.ts      # 实现
└── instance-manager.test.ts # 单元测试
```

#### 集成测试（集中管理）

- **位置**：`tests/integration/` 目录
- **命名**：`{feature}.test.ts`
- **范围**：多个模块协作测试
- **特点**：验证模块间交互

#### 端到端测试（集中管理）

- **位置**：`tests/e2e/` 目录
- **命名**：`{scenario}.test.ts`
- **范围**：完整业务流程测试
- **特点**：模拟真实用户场景

### 8.2 错误处理策略

#### 8.2.1 错误分类

```typescript
// 领域错误 - 业务规则违反
class DomainError extends Error {
  constructor(message: string, public readonly code: string) {
    super(message);
  }
}

// 具体错误类型
class InstanceNotFoundError extends DomainError {
  constructor(id: number) {
    super(`实例ID ${id} 不存在`, 'INSTANCE_NOT_FOUND');
  }
}

class InvalidCommandError extends DomainError {
  constructor(message: string) {
    super(message, 'INVALID_COMMAND');
  }
}

class NonInteractiveError extends DomainError {
  constructor() {
    super('该实例为非交互式，不支持接收手动指令', 'NON_INTERACTIVE');
  }
}
```

#### 8.2.2 错误处理原则

1. **早失败**：在入口层尽早验证输入，避免无效数据流入深层
2. **错误转换**：底层错误转换为领域错误，避免暴露实现细节
3. **用户友好**：错误消息包含「原因 + 解决方案/示例」
4. **统一处理**：应用层统一捕获并格式化错误响应

#### 8.2.3 异常提示模板

| 场景 | 提示模板 |
|------|----------|
| ID不存在 | `实例ID不存在！可通过 /list 查看运行中的实例` |
| 格式错误 | `格式错误！正确示例：{示例命令}` |
| 类型不支持 | `不支持的类型！当前支持：{类型列表}` |
| 非交互式 | `该实例为非交互式，不支持接收手动指令` |
| 未设置默认 | `未指定实例！可通过 /use ID 设置默认，或 /ID 内容 直接指定` |
| 默认失效 | `默认实例已失效！请重新通过 /use ID 切换` |

### 8.3 性能约束

| 指标 | 约束 | 说明 |
|------|------|------|
| 命令响应时间 | < 100ms | 纯内存操作应在100ms内完成 |
| 实例创建时间 | < 500ms | 进程创建应在500ms内完成 |
| 消息转发延迟 | < 50ms | 消息透传应在50ms内完成 |
| 最大实例数 | 可配置 | 默认限制50个并发实例 |
| 日志保留数 | 可配置 | 默认保留最近100条 |

## 9. 可测试性设计

### 9.1 依赖注入规范

1. **构造函数注入**：所有依赖通过构造函数传入
```typescript
// ✅ 推荐
class InstanceManager {
  constructor(
    private readonly terminalAdapter: ITerminalAdapter,
    private readonly logStorage: ILogStorage
  ) {}
}

// ❌ 避免
class InstanceManager {
  private terminalAdapter = new TerminalAdapter(); // 内部实例化
}
```

2. **接口依赖**：依赖接口而非具体类
```typescript
// ✅ 推荐
constructor(adapter: ITerminalAdapter) {}

// ❌ 避免
constructor(adapter: TerminalAdapter) {}
```

### 9.2 接口隔离原则

1. **小接口优于大接口**：
```typescript
// ✅ 推荐 - 拆分为小接口
interface IReadable {
  read(): string;
}

interface IWritable {
  write(data: string): void;
}

// ❌ 避免 - 大接口
interface IBigInterface {
  read(): string;
  write(data: string): void;
  open(): void;
  close(): void;
  // ... 更多方法
}
```

2. **单一职责**：每个接口只定义一个职责领域

### 9.3 测试替身策略

| 类型 | 使用场景 | 实现方式 |
|------|----------|----------|
| Stub | 提供固定返回值 | 硬编码返回 |
| Mock | 验证交互行为 | 记录调用次数、参数 |
| Fake | 简化实现（如内存存储） | 轻量级实现 |
| Spy | 部分真实+部分监控 | 包装真实对象 |

示例：
```typescript
// Mock 终端适配器
class MockTerminalAdapter implements ITerminalAdapter {
  spawnCalls: { type: string; params: string[] }[] = [];
  
  async spawn(type: string, params: string[]): Promise<IInstanceProcess> {
    this.spawnCalls.push({ type, params });
    return new MockInstanceProcess();
  }
}

// Fake 日志存储
class FakeLogStorage implements ILogStorage {
  private logs: InstanceLog[] = [];
  
  save(log: InstanceLog): void {
    this.logs.push(log);
  }
  
  getAll(): InstanceLog[] {
    return this.logs;
  }
}
```

### 9.4 测试分层策略

| 测试类型 | 比例 | 范围 | 执行时间 |
|----------|------|------|----------|
| 单元测试 | 70% | 单个类/函数 | < 10ms |
| 集成测试 | 20% | 模块间交互 | < 100ms |
| E2E测试 | 10% | 完整业务流程 | < 1s |

#### 测试金字塔

```
         /\
        /  \
       / E2E\      少量（10%）- 验证完整流程
      /────────\n     /          \
    /  集成测试  \   中等（20%）- 验证模块协作
   /──────────────\
  /                \
 /     单元测试      \ 大量（70%）- 验证业务逻辑
/────────────────────\
```

#### 各层测试重点

- **单元测试**：验证业务逻辑正确性，Mock所有外部依赖
- **集成测试**：验证模块间协作，使用 Fake 替代外部服务
- **E2E测试**：验证完整用户场景，使用真实依赖（如测试飞书机器人）
