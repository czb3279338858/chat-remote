# 移动端聊天控制多类型实例系统 —— 需求规格说明书

## 1. 项目概述

### 1.1 项目背景
通过手机聊天软件（飞书），统一管理 PC 上多种交互式/非交互式实例，实现指令简洁、异常提示明确、全程无鉴权、纯透传交互。

### 1.2 核心目标

| 目标 | 说明 |
|------|------|
| **统一管理** | 通过手机端统一控制 PC 上的多种类型实例 |
| **简洁交互** | 指令简洁明了，易于记忆和使用 |
| **明确反馈** | 异常提示包含原因和解决方案 |
| **纯透传** | 消息不做业务处理，直接转发给目标实例 |
| **无鉴权** | 全程无需身份验证，简化使用流程 |

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| 实例（Instance） | 被管理的运行实体，如 qodercli 等进程 |
| 实例类型（Type） | 实例的分类标识，每种类型有特定的启动参数和运行模式 |
| 交互式实例 | 启动后持续运行，支持多次接收输入、返回输出的实例 |
| 非交互式实例 | 启动后自动运行、输出结果、自动结束的实例 |
| 运行状态 | 实例的当前状态：运行中（空闲/忙碌）、已退出 |
| 默认实例 | 通过 `/use ID` 设置的当前操作目标实例 |
| 可中断特性 | 实例支持在执行过程中被 `/interrupt` 指令打断 |

## 2. 实例模型

### 2.1 实例属性

| 字段 | 类型 | 说明 |
|------|------|------|
| ID | number | 自增数字，唯一、不可修改，作为实例核心标识 |
| 类型 | string | 如 qodercli 等 |
| 类型说明 | string | 简短描述该类型用途（展示在 `/help` 中） |
| 用户备注 | string | 自定义用途标签，支持后期修改 |
| 启动参数 | string[] | 各类型实例自定义，创建时传入 |
| 运行模式 | RunMode | 交互式(INTERACTIVE) / 非交互式(NON_INTERACTIVE) |
| 运行状态 | InstanceStatus | 运行中-空闲(IDLE) / 运行中-忙碌(BUSY) / 已退出(EXITED) |
| 实例特性 | Feature | 可中断(INTERRUPTIBLE) / 不可中断(NON_INTERRUPTIBLE) |
| 启动时间 | Date | 实例创建时间 |

### 2.2 实例类型定义

| 实例类型 | 运行模式 | 是否可中断 | 典型用途 | 启动参数示例 |
|----------|----------|-----------|----------|--------------|
| qodercli | 交互式 | **否** | 大语言模型 CLI 对话 | `qodercli`、`qodercli -w=/path` |

## 3. 运行模式规则

### 3.1 交互式实例

**典型代表**：qodercli

#### 运行规则

1. 启动后持续运行，支持多次接收输入、返回输出
2. 需手动 `/kill ID` 关闭，或异常退出
3. 运行中始终显示在 `/list` 中
4. **响应分块机制**（因聊天软件字数限制）：
   - 实例进程输出响应内容，系统持续接收
   - 当累积达到预设长度阈值时，向聊天软件发送消息
   - 每条消息标明顺序号（如 `[1]`、`[2]`），最后一条标 `[完成]`
5. **中断能力（按类型定义）**：部分实例类型支持 `/interrupt` 指令，可中断当前处理并返回最新状态

#### 状态流转

```
不存在 → /new → 运行中-空闲 → send → 运行中-忙碌 → 响应分块发送 → 运行中-空闲
                                           ↓
                              /kill 或异常 → 已退出 → /log
```

### 3.2 非交互式实例

#### 运行规则

1. 启动 → 自动运行 → 输出全部结果 → 自动结束
2. 运行过程中不接受任何外部输入（`/ID 内容`/纯文本均拒绝）
3. 结束后自动从 `/list` 移除，记入 `/log`（退出原因标记为"正常结束"）

#### 状态流转

```
不存在 → /new → 运行中-忙碌（自动运行）→ 自动结束 → 已退出 → /log
                      ↑
                   拒绝send
```

### 4.1 指令速查表

| 指令 | 格式 | 功能描述 |
|------|------|----------|
| `/help` | `/help [指令名] [类型名]` | 显示帮助信息 |
| `/new` | `/new <类型> <备注> [参数...]` | 创建新实例 |
| `/list` | `/list` | 列出运行中实例 |
| `/log` | `/log` | 查看历史记录 |
| `/rename` | `/rename <ID> <新备注>` | 修改实例备注 |
| `/use` | `/use <ID>` | 设置默认实例 |
| `/ID` | `/<ID> <内容>` | 向指定实例发送消息 |
| 纯文本 | `<任意文本内容>` | 发送给默认实例 |
| `/kill` | `/kill <ID>` | 关闭指定实例 |
| `/killall` | `/killall [类型]` | 批量关闭实例 |
| `/interrupt` | `/interrupt [ID]` | 中断实例执行 |

### 4.2 指令详细规格

#### 4.2.1 `/help [指令名] [类型名]`

**功能**：显示帮助信息，支持查询指定指令的详细说明；对于 `/new` 指令，支持查询指定类型的具体使用方式

**格式**：
- `/help` - 显示所有指令列表
- `/help <指令名>` - 显示指定指令的详细说明
- `/help new <类型名>` - 显示 `/new` 指令中指定类型的具体使用方式

**示例**：
```
/help                 - 显示所有指令列表
/help new             - 显示 /new 指令的详细说明
/help new qodercli    - 显示 /new 指令中 qodercli 类型的具体使用方式
/help list            - 显示 /list 指令的详细说明
```

**输出示例**：

```
📖 帮助信息

用法：/help [指令名]

  /help           - 显示所有指令列表
  /help <指令名>  - 显示指定指令的详细说明

📋 指令列表

/new <类型> <备注> [参数]   - 创建新实例
/list                       - 列出运行中实例
/log                        - 查看历史记录
/use <ID>                   - 设置默认实例
/<ID> <内容>                - 向指定实例发送消息
/kill <ID>                  - 关闭指定实例
/killall [类型]             - 批量关闭实例
/interrupt [ID]             - 中断实例执行
/rename <ID> <新备注>       - 修改实例备注
/help [指令名]              - 显示帮助信息

💡 使用 /help <指令名> 查看详细说明，如 /help new
```

**异常场景**：

| 异常类型 | 触发条件 | 提示信息 |
|----------|----------|----------|
| 指令不存在 | 指定的指令名不存在 | `未知指令：xxx。使用 /help 查看所有指令` |
| 类型不存在 | `/help new <类型>` 中的类型不存在 | `未知类型：xxx。当前支持：qodercli` |

#### 4.2.2 `/new <类型> <备注> [参数...]`

**功能**：创建指定类型实例，参数由对应类型定义

**参数说明**：
- `类型`：实例类型标识（当前仅支持 qodercli）
- `备注`：用户自定义的实例标签（用于识别用途）
- `参数`：类型特定的启动参数（可选，根据类型需求）

**示例**：
```
/new qodercli 代码助手
/new qodercli 项目助手 -w=/path/to/project
/new qodercli 安全模式 --allowed-tools=READ
```

**响应流程**：
1. 系统创建实例进程，分配 ID
2. 实例进入**运行中-忙碌**状态（启动中/初始化中）
3. 捕获实例的初始输出（如启动日志、欢迎信息等）
4. 将初始输出随创建成功消息一并返回
5. 当实例完成初始化、准备好接收输入时，自动转为**运行中-空闲**状态

**成功响应示例**：
```
✅ 实例创建成功

[ID:1] qodercli | 代码助手 | 运行中-忙碌

[1] 正在初始化 qodercli...
[2] 加载项目配置...
[3] 准备就绪，等待输入
[完成]

实例已进入空闲状态，可以发送消息
```

**异常场景**：

| 异常类型 | 触发条件 | 提示信息 |
|----------|----------|----------|
| 格式错误 | 缺失「类型」或「备注」 | `格式错误！正确示例：/new qodercli 代码助手` |
| 类型不支持 | 类型不在支持列表中 | `不支持的类型！当前支持：qodercli` |
| 参数错误 | 参数格式不符合类型要求 | `参数错误！该类型参数示例：/new qodercli 助手 -w=/path` |

#### 4.2.11 `/interrupt [ID]`

**功能**：向指定交互式实例发送中断信号

**适用场景**：如大语言模型 CLI 正在生成长回复，用户希望打断并重新输入

**参数说明**：
- `ID`（可选）：目标实例的数字标识，省略时使用默认实例

**示例**：
```
/interrupt 1   - 中断 ID 为 1 的实例
/interrupt     - 中断默认实例
```

**异常场景**：

| 异常类型 | 触发条件 | 提示信息 |
|----------|----------|----------|
| ID不存在 | 指定ID不存在或已退出 | `实例ID不存在！可通过 /list 查看运行中的实例` |

| 非交互式 | 目标实例为非交互式 | `该实例为非交互式，不支持中断操作` |
| 不支持中断 | 实例类型不支持中断 | `该实例类型不支持中断操作` |

## 5. 全局规则

### 5.1 异常提示规范

1. **错误原因 + 解决方案**：所有指令异常提示需包含「错误原因 + 解决方案/示例」，而非仅告知"错误"
2. **统一模板**：使用统一的异常提示模板，确保用户体验一致性

### 5.2 实例生命周期规则

1. **非交互式实例限制**：非交互式实例一律拒绝接收任何手动输入（`/ID 内容` 或纯文本消息）
2. **自动清理**：实例退出后自动从 `/list` 移除，并记录退出时间、原因至 `/log`
3. **状态同步**：实例状态变化实时同步到 `/list` 显示

### 5.3 交互规则

1. **无鉴权**：全程无鉴权、无用户区分，所有操作纯透传
2. **指令容错**：指令支持基础容错：
   - 多空格自动合并
   - 大小写自动转为小写

### 5.4 异常提示模板汇总

| 场景 | 提示模板 |
|------|----------|
| ID不存在 | `实例ID不存在！可通过 /list 查看运行中的实例` |
| 格式错误 | `格式错误！正确示例：{示例命令}` |
| 类型不支持 | `不支持的类型！当前支持：{类型列表}` |
| 非交互式 | `该实例为非交互式，不支持接收手动指令` |
| 未设置默认 | `未指定实例！可通过 /use ID 设置默认，或 /ID 内容 直接指定` |
| 默认失效 | `默认实例已失效！请重新通过 /use ID 切换` |
| 不支持中断 | `该实例类型不支持中断操作` |

## 6. 非功能性需求

### 6.1 性能要求

| 指标 | 约束 | 说明 |
|------|------|------|
| 最大实例数 | 可配置 | 默认限制50个并发实例 |
| 日志保留数 | 可配置 | 默认保留最近100条 |

### 6.2 可用性要求

1. **容错性**：系统应能优雅处理用户输入错误，提供清晰的错误提示
2. **一致性**：相同操作在不同场景下应产生一致的结果
3. **可恢复性**：实例异常退出时，应正确记录日志并清理资源

### 6.3 扩展性要求

1. **类型扩展**：系统架构预留实例类型扩展能力
   - 当前仅实现 `qodercli` 类型
   - 后续可扩展其他类型（如 ssh、node、python 等）
   - 新增类型需定义：运行模式、中断特性、启动参数解析、响应处理逻辑
   - 类型定义集中管理，便于后续扩展

2. **命令扩展**：支持注册新的指令处理器

3. **平台扩展**：消息接入层可适配不同聊天平台（微信、钉钉等）
